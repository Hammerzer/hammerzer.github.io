## 问题：A*做路径规划的时候，可选方向有哪些？

### 1 平面规划

#### （1）正方形网格（最常用，如栅格地图）

适用于二维平面的离散化场景（如游戏地图、室内机器人导航），根据移动是否允许对角线方向，分为两种：

1. **四方向（4 - 连通）**：仅允许水平和垂直移动，方向为「上、下、左、右」。
   - 特点：路径由直角转折组成，无斜向移动，适合对移动方向有严格限制的场景（如轮式机器人无法斜向行驶）。
2. **八方向（8 - 连通）**：在四方向基础上增加对角线移动，方向为「上、下、左、右、左上、右上、左下、右下」。
   - 特点：路径更灵活，可能存在斜向捷径，但需注意对角线移动的代价计算（通常为√2 倍的横向 / 纵向单位代价，避免算法优先选择不合理的斜向路径）。

#### （2）六边形网格（6 - 连通）

适用于需要更均匀距离度量的场景（如无人机区域覆盖、策略游戏），每个节点有 6 个相邻方向（相邻六边形的中心连线方向）。

- 特点：各方向距离均等（无正方形网格中「边距≠对角线距」的问题），方向对称性更好，路径平滑度优于正方形网格。

#### （3）连续空间（离散化方向）

在非网格的连续空间中（如室外大范围路径规划），A * 通常会将方向**离散化为若干角度**（避免无限方向导致的计算爆炸），例如：

- 按固定角度间隔划分（如每 15°、30° 或 45° 一个方向），形成 8 方向、12 方向、24 方向等。
- 方向数量越多，路径越接近最优连续路径，但搜索复杂度也会增加。

#### （4）带约束的方向（如移动方向限制）

在实际场景中，可能因障碍物、机器人运动学约束（如履带式机器人转向受限）进一步限制方向，例如：

- 仅允许「前、后、左前 45°、右前 45°」等有限方向；
- 方向需与当前运动方向关联（如只能向当前方向的 ±30° 范围内转向）。

### 2 三维规划

无人机路径规划在三维空间中的方向设计，需结合其**三维运动特性**（可沿 x、y、z 轴自由移动）和**物理约束**（如最小转弯半径、爬升 / 俯冲角限制等），常见的方向划分方式主要有以下几类：

#### （1）三维网格离散化方向（基于立方体网格模型）

适用于将三维空间离散为立方体栅格（如室内 / 城市复杂环境避障），方向设计以 “当前栅格节点到相邻栅格节点的移动路径” 为基准，常见有 3 种：

- **6 方向（6 - 连通）**：仅允许沿三维坐标轴的正负方向移动，即「±x（前后）、±y（左右）、±z（上下）」。
  - 特点：移动路径为轴对齐的直线，方向简单直观，计算量小；但路径可能较 “生硬”（多直角转折），适合对算力有限制或运动方向受限的小型无人机（如室内微型无人机）。

- **12 方向（12 - 连通）**：在 6 方向基础上，增加 “面对角线” 方向，即沿立方体每个面的对角线移动（如 x+y、x-y、x+z、x-z、y+z、y-z，及对应的反方向）。
  - 特点：方向更灵活，路径可包含斜向移动（如在同一高度面内斜向飞行，或同时改变水平位置和高度），比 6 方向更接近无人机实际运动轨迹，但需注意对角线移动的代价计算（如距离为√2 倍轴长，避免路径代价失真）。

- **26 方向（26 - 连通）**：包含立方体所有相邻节点的移动方向，除 6 个轴方向、12 个面对角线方向外，再增加 8 个 “空间对角线” 方向（如 x+y+z、x+y-z 等）。
  - 特点：方向覆盖最全面，路径灵活性最高，能适应复杂三维障碍物环境（如密集建筑群、峡谷）；但方向数量多，会增加 A * 等算法的搜索复杂度，适合对路径平滑度要求高的场景（如户外中型无人机）。

#### （2）球面坐标角度离散化方向（连续空间近似）

适用于大范围三维空间（如室外空旷区域、低空飞行），不依赖栅格划分，而是将方向按**球面坐标（方位角 θ、俯仰角 φ）** 离散为有限角度组合：

- **方位角 θ**：绕 z 轴（竖直方向）的旋转角度（0°~360°），通常按固定间隔划分（如每 15°、30° 或 45° 一个方向）；
- **俯仰角 φ**：与水平面的夹角（-90°~90°，负角为俯冲，正角为爬升），通常按无人机物理性能限制划分（如 ±15°、±30°，避免过大仰角导致失速）。

例如：方位角每 30° 划分（12 个方向），俯仰角每 15° 划分（7 个方向），可形成 12×7=84 个方向。

特点：方向精度可灵活调整（角度间隔越小，方向越密集，路径越接近最优连续路径），但计算量随方向数量呈指数增长，需在精度与效率间平衡。

#### （3）带运动学约束的方向（贴合无人机物理特性）

实际无人机受自身动力、机动性限制（如最小转弯半径、最大角速度），方向选择需满足运动学约束，常见有：

- **基于 “航向角 - 俯仰角” 的有限方向集**：

限制无人机的转向角度（如相邻路径段的航向角变化不超过 30°，俯仰角变化不超过 15°），方向仅在当前航向的 “允许转向范围内” 选取（如当前航向为 0°，则下一个航向只能在 - 30°~30° 范围内）。

- **“速度矢量” 方向约束**：

无人机的运动方向需与速度矢量一致，且速度大小在合理范围内（如最小速度 v_min、最大速度 v_max），方向变化需满足 “加速度不超过最大允许值”（避免过载）。

特点：方向设计更贴近无人机实际飞行能力，规划出的路径可直接用于控制执行（无需后续平滑处理），适合工程化应用（如物流无人机、植保无人机）。

## 问题：定时器设计与优化

每30s刷新一波小兵，游戏开始5分钟刷新Boss（单次任务），支持随时取消未触发的boss任务。

（1）请设计定时节点的数据结构

（2）选择有序数组或者最小堆实现定时器节点的添加

（3）实现定时器逻辑，可用伪代码实现

## AI回答

### （1）定时节点的数据结构设计

定时节点需要包含任务的核心信息：触发时间、任务类型、重复属性、唯一标识（用于取消）及执行逻辑。设计如下：

```cpp
struct TimerNode {
    int64_t trigger_time;       // 触发时间（绝对时间戳，单位：毫秒）
    int task_id;                // 任务唯一ID（用于取消任务）
    bool is_repeat;             // 是否重复任务（true：重复，false：单次）
    int repeat_interval;        // 重复间隔（单位：毫秒，仅repeat为true时有效）
    enum TaskType {             // 任务类型
        MINION_REFRESH,         // 小兵刷新
        BOSS_REFRESH            // BOSS刷新
    } task_type;
    std::function<void()> callback;  // 任务执行的回调函数（如刷新小兵/BOSS的逻辑）
};
```

**核心字段说明**：

- `trigger_time`：用绝对时间戳统一管理，避免相对时间计算误差；
- `task_id`：唯一标识任务，支持精准取消（尤其是 BOSS 这类单次任务）；
- `is_repeat` 和 `repeat_interval`：区分单次 / 重复任务（如小兵每 30s 刷新，repeat_interval=30000ms）；
- `callback`：封装任务执行逻辑，解耦定时器与业务。

### （2）选择最小堆实现定时器节点的添加

**选择最小堆而非有序数组的原因**：

- **插入效率**：最小堆插入的时间复杂度为 `O(log n)`，而有序数组插入需移动元素，复杂度为 `O(n)`，在任务数量较多时堆更高效；
- **获取最近触发任务**：两者均能 `O(1)` 获取（堆顶 / 数组首元素），但堆的整体操作更均衡；
- **支持取消任务**：结合哈希表（`task_id -> 堆中节点索引`），可将取消操作优化为 `O(log n)`（找到节点后执行堆的删除操作），而有序数组取消需 `O(n)` 查找 + 删除。

### （3）定时器逻辑实现（伪代码）

定时器核心逻辑：维护最小堆管理任务，循环处理到期任务，支持动态添加 / 取消任务。

```cpp
#include <vector>
#include <unordered_map>
#include <functional>
#include <chrono>
#include <thread>
#include <iostream>
#include <stdint.h>  // 用于int64_t

// 任务类型枚举
enum class TaskType {
    MINION_REFRESH,  // 小兵刷新
    BOSS_REFRESH     // BOSS刷新
};

// 定时任务节点
struct TimerNode {
    int64_t trigger_time;       // 触发时间（绝对时间戳，单位：毫秒）
    int task_id;                // 任务唯一ID（用于取消）
    bool is_repeat;             // 是否重复任务
    int repeat_interval;        // 重复间隔（毫秒，仅重复任务有效）
    TaskType task_type;         // 任务类型
    std::function<void()> callback;  // 任务执行逻辑
};

// 定时器管理器（核心类）
class TimerManager {
    private:
    std::vector<TimerNode> min_heap;  // 最小堆：按trigger_time升序
    std::unordered_map<int, size_t> id_map;  // task_id -> 堆中索引（快速查找）

    // 工具函数：获取当前时间戳（毫秒）
    int64_t get_current_time() const {
        auto now = std::chrono::system_clock::now();
        auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(
            now.time_since_epoch()
        );
        return ms.count();
    }

    // 堆调整：上浮（维护最小堆）
    void heapify_up(size_t idx) {
        while (idx > 0) {
            size_t parent = (idx - 1) / 2;
            if (min_heap[idx].trigger_time >= min_heap[parent].trigger_time) {
                break;  // 父节点更小，堆序正确
            }
            // 交换父子节点
            std::swap(min_heap[idx], min_heap[parent]);
            // 更新哈希表索引
            id_map[min_heap[idx].task_id] = idx;
            id_map[min_heap[parent].task_id] = parent;
            idx = parent;
        }
    }

    // 堆调整：下沉（维护最小堆）
    void heapify_down(size_t idx) {
        size_t heap_size = min_heap.size();
        while (true) {
            size_t left = 2 * idx + 1;
            size_t right = 2 * idx + 2;
            size_t smallest = idx;

            if (left < heap_size && min_heap[left].trigger_time < min_heap[smallest].trigger_time) {
                smallest = left;
            }
            if (right < heap_size && min_heap[right].trigger_time < min_heap[smallest].trigger_time) {
                smallest = right;
            }
            if (smallest == idx) {
                break;  // 子节点均不更小，堆序正确
            }
            // 交换节点
            std::swap(min_heap[idx], min_heap[smallest]);
            // 更新哈希表索引
            id_map[min_heap[idx].task_id] = idx;
            id_map[min_heap[smallest].task_id] = smallest;
            idx = smallest;
        }
    }

    public:
    TimerManager() = default;

    // 添加任务
    void add_task(const TimerNode& task) {
        // 检查task_id唯一性（实际场景需确保调用者传入唯一ID）
        if (id_map.count(task.task_id)) {
            std::cerr << "Task ID " << task.task_id << " already exists!\n";
            return;
        }
        // 插入堆尾
        min_heap.push_back(task);
        size_t idx = min_heap.size() - 1;
        id_map[task.task_id] = idx;
        // 上浮调整堆
        heapify_up(idx);
    }

    // 取消任务（通过task_id）
    bool cancel_task(int task_id) {
        if (!id_map.count(task_id)) {
            return false;  // 任务不存在
        }
        // 获取任务在堆中的索引
        size_t idx = id_map[task_id];
        size_t last_idx = min_heap.size() - 1;

        // 用最后一个元素覆盖当前元素
        min_heap[idx] = min_heap[last_idx];
        id_map[min_heap[idx].task_id] = idx;  // 更新最后元素的索引

        // 删除最后一个元素
        min_heap.pop_back();
        id_map.erase(task_id);

        // 调整堆（根据情况选择下沉或上浮）
        if (idx < min_heap.size()) {
            heapify_down(idx);
            heapify_up(idx);
        }
        return true;
    }

    // 启动定时器主循环
    void run() {
        while (true) {
            if (min_heap.empty()) {
                // 无任务时休眠100ms，避免空轮询
                std::this_thread::sleep_for(std::chrono::milliseconds(100));
                continue;
            }

            // 获取堆顶任务（最早触发的任务）
            TimerNode top = min_heap[0];
            int64_t current = get_current_time();
            int64_t wait_ms = top.trigger_time - current;

            if (wait_ms > 0) {
                // 未到触发时间，休眠至触发时间
                std::this_thread::sleep_for(std::chrono::milliseconds(wait_ms));
            }

            // 执行任务
            top.callback();

            // 处理任务后续逻辑
            if (top.is_repeat) {
                // 重复任务：更新触发时间后重新加入堆
                cancel_task(top.task_id);  // 先删除原任务
                TimerNode new_task = top;
                new_task.trigger_time = get_current_time() + top.repeat_interval;  // 基于当前时间更新
                add_task(new_task);
            } else {
                // 单次任务：直接删除
                cancel_task(top.task_id);
            }
        }
    }
};
```

## 第二题：实现高精度定时器模块

在 Linux 环境下，基于 `timefd` 和 `epoll` 可以实现高精度定时器模块（支持微秒 / 纳秒级精度）。`timefd` 是 Linux 提供的将时间事件抽象为文件描述符的机制，配合 `epoll` 的 I/O 多路复用能力，可高效监听超时事件，避免传统定时器的空轮询损耗。以下是具体实现：

### 核心设计思路

1. **高精度时间源**：使用 `timefd` 作为时间触发源，其基于 Linux 内核时钟（`CLOCK_MONOTONIC`，单调递增，不受系统时间调整影响），支持纳秒级精度。
2. **任务管理**：延续最小堆存储定时任务，快速获取最近触发的任务；用哈希表映射 `task_id` 与堆索引，支持快速取消。
3. **事件驱动**：通过 `epoll` 监听 `timefd` 的读事件，当超时时间到达时，`epoll` 唤醒进程处理任务，避免无效轮询。
4. **动态更新**：每次任务处理后，根据新的堆顶任务更新 `timefd` 的超时时间，确保下一次触发精准。

### 实现代码

```cpp
#include <vector>
#include <unordered_map>
#include <functional>
#include <sys/epoll.h>
#include <sys/timerfd.h>
#include <unistd.h>
#include <cstdint>
#include <chrono>
#include <mutex>
#include <iostream>
#include <stdexcept>
#include <algorithm>

// 任务类型枚举
enum class TaskType {
    MINION_REFRESH,  // 小兵刷新
    BOSS_REFRESH     // BOSS刷新
};

// 定时任务节点（高精度版）
struct TimerNode {
    int64_t trigger_time_ns;    // 触发时间（绝对时间戳，单位：纳秒）
    int task_id;                // 任务唯一ID
    bool is_repeat;             // 是否重复任务
    int64_t repeat_interval_ns; // 重复间隔（纳秒，仅重复任务有效）
    TaskType task_type;         // 任务类型
    std::function<void()> callback;  // 任务执行逻辑
};

// 高精度定时器管理器
class HighResTimerManager {
private:
    std::vector<TimerNode> min_heap;  // 最小堆（按trigger_time_ns升序）
    std::unordered_map<int, size_t> id_map;  // task_id -> 堆索引
    int epoll_fd;  // epoll实例句柄
    int time_fd;   // timefd句柄
    std::mutex mtx;  // 保护堆和哈希表的线程锁

    // 获取当前单调时间（纳秒，不受系统时间调整影响）
    int64_t get_current_time_ns() const {
        struct timespec ts;
        if (clock_gettime(CLOCK_MONOTONIC, &ts) == -1) {
            throw std::runtime_error("clock_gettime failed");
        }
        return (int64_t)ts.tv_sec * 1000000000 + ts.tv_nsec;
    }

    // 堆调整：上浮
    void heapify_up(size_t idx) {
        while (idx > 0) {
            size_t parent = (idx - 1) / 2;
            if (min_heap[idx].trigger_time_ns >= min_heap[parent].trigger_time_ns) {
                break;
            }
            std::swap(min_heap[idx], min_heap[parent]);
            id_map[min_heap[idx].task_id] = idx;
            id_map[min_heap[parent].task_id] = parent;
            idx = parent;
        }
    }

    // 堆调整：下沉
    void heapify_down(size_t idx) {
        size_t heap_size = min_heap.size();
        while (true) {
            size_t left = 2 * idx + 1;
            size_t right = 2 * idx + 2;
            size_t smallest = idx;

            if (left < heap_size && min_heap[left].trigger_time_ns < min_heap[smallest].trigger_time_ns) {
                smallest = left;
            }
            if (right < heap_size && min_heap[right].trigger_time_ns < min_heap[smallest].trigger_time_ns) {
                smallest = right;
            }
            if (smallest == idx) {
                break;
            }
            std::swap(min_heap[idx], min_heap[smallest]);
            id_map[min_heap[idx].task_id] = idx;
            id_map[min_heap[smallest].task_id] = smallest;
            idx = smallest;
        }
    }

    // 更新timefd的超时时间（基于堆顶任务）
    void update_timefd() {
        std::lock_guard<std::mutex> lock(mtx);
        if (min_heap.empty()) {
            // 无任务时，禁用timefd（设置为永久超时）
            struct itimerspec new_value = {0};
            timerfd_settime(time_fd, 0, &new_value, nullptr);
            return;
        }

        // 堆顶任务的触发时间
        int64_t target_ns = min_heap[0].trigger_time_ns;
        int64_t now_ns = get_current_time_ns();
        int64_t delay_ns = std::max(int64_t(0), target_ns - now_ns);  // 确保非负

        // 设置timefd超时时间（绝对时间）
        struct itimerspec new_value;
        new_value.it_interval = {0, 0};  // 仅一次超时
        new_value.it_value.tv_sec = delay_ns / 1000000000;
        new_value.it_value.tv_nsec = delay_ns % 1000000000;

        if (timerfd_settime(time_fd, 0, &new_value, nullptr) == -1) {
            throw std::runtime_error("timerfd_settime failed");
        }
    }

public:
    HighResTimerManager() {
        // 初始化epoll
        epoll_fd = epoll_create1(EPOLL_CLOEXEC);
        if (epoll_fd == -1) {
            throw std::runtime_error("epoll_create1 failed");
        }

        // 初始化timefd（CLOCK_MONOTONIC保证时间单调递增）
        time_fd = timerfd_create(CLOCK_MONOTONIC, TFD_NONBLOCK | TFD_CLOEXEC);
        if (time_fd == -1) {
            close(epoll_fd);
            throw std::runtime_error("timerfd_create failed");
        }

        // 将timefd添加到epoll监听（水平触发，读事件）
        struct epoll_event ev;
        ev.events = EPOLLIN;
        ev.data.fd = time_fd;
        if (epoll_ctl(epoll_fd, EPOLL_CTL_ADD, time_fd, &ev) == -1) {
            close(time_fd);
            close(epoll_fd);
            throw std::runtime_error("epoll_ctl failed");
        }
    }

    ~HighResTimerManager() {
        close(time_fd);
        close(epoll_fd);
    }

    // 添加任务（线程安全）
    void add_task(const TimerNode& task) {
        std::lock_guard<std::mutex> lock(mtx);
        if (id_map.count(task.task_id)) {
            std::cerr << "Task ID " << task.task_id << " already exists!\n";
            return;
        }
        // 插入堆并调整
        min_heap.push_back(task);
        size_t idx = min_heap.size() - 1;
        id_map[task.task_id] = idx;
        heapify_up(idx);
        // 若新任务是堆顶，更新timefd
        if (idx == 0) {
            update_timefd();
        }
    }

    // 取消任务（线程安全）
    bool cancel_task(int task_id) {
        std::lock_guard<std::mutex> lock(mtx);
        if (!id_map.count(task_id)) {
            return false;
        }
        size_t idx = id_map[task_id];
        size_t last_idx = min_heap.size() - 1;

        // 用最后一个元素覆盖当前元素
        min_heap[idx] = min_heap[last_idx];
        id_map[min_heap[idx].task_id] = idx;

        // 删除最后一个元素
        min_heap.pop_back();
        id_map.erase(task_id);

        // 调整堆
        if (idx < min_heap.size()) {
            heapify_down(idx);
            heapify_up(idx);
        }

        // 若堆顶变化，更新timefd
        update_timefd();
        return true;
    }

    // 启动定时器主循环（阻塞，处理事件）
    void run() {
        const int MAX_EVENTS = 10;
        struct epoll_event events[MAX_EVENTS];

        while (true) {
            // 等待epoll事件（阻塞，直到timefd超时或被信号中断）
            int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
            if (nfds == -1) {
                if (errno == EINTR) continue;  // 被信号中断，继续等待
                throw std::runtime_error("epoll_wait failed");
            }

            // 处理事件（仅关注timefd的读事件）
            for (int i = 0; i < nfds; ++i) {
                if (events[i].data.fd == time_fd && (events[i].events & EPOLLIN)) {
                    // 读取timefd的值（清除事件，否则会一直触发）
                    uint64_t exp;
                    read(time_fd, &exp, sizeof(exp));

                    // 处理所有已超时的任务
                    process_expired_tasks();
                }
            }
        }
    }

    // 处理所有已超时的任务
    void process_expired_tasks() {
        std::lock_guard<std::mutex> lock(mtx);
        int64_t now_ns = get_current_time_ns();

        // 循环处理堆顶已超时的任务
        while (!min_heap.empty() && min_heap[0].trigger_time_ns <= now_ns) {
            TimerNode top = min_heap[0];
            // 执行任务回调（注意：若回调耗时较长，可考虑异步执行）
            top.callback();

            // 移除当前堆顶任务
            cancel_task(top.task_id);  // 内部已加锁，此处lock仍有效

            // 处理重复任务：更新时间后重新添加
            if (top.is_repeat) {
                TimerNode new_task = top;
                new_task.trigger_time_ns = now_ns + top.repeat_interval_ns;  // 基于当前时间更新
                add_task(new_task);  // 内部已加锁
            }
        }

        // 更新timefd为下一个任务的超时时间
        update_timefd();
    }
};
```

## 3.1 递归函数的时间复杂度

核心是分析递归调用的次数以及每次递归中基本操作的执行次数。以下是常用的分析方法和步骤：

**主方法（Master Theorem）**：针对形如 `(T(n) = aT(n/b) + f(n))` 的递推关系，直接套用公式（适用于分治类递归）。

**1、建立递推公式**

- 设 \(T(n)\) 为问题规模为 n 时的时间复杂度。
- 若每次递归将问题分解为 a 个规模为 \(n/b\) 的子问题，且合并子问题的时间为 \(f(n)\)，则递推公式为：`(T(n) = aT(n/b) + f(n))`
- 例：斐波那契递归 \(F(n) = F(n-1) + F(n-2)\)，递推公式为 `T(n) = T(n-1) + T(n-2) + O(1)`。

**2、求解递推公式**

- 递归树法：

  画出递归树，每层节点数为 \(a^k\)（k 为层数），每层子问题规模为 \(n/b^k)，直到规模为 1（终止条件）。总时间为所有层的操作数之和。

  例：归并排序 \(T(n) = 2T(n/2) + O(n)\)，递归树每层操作数均为 \(O(n)\)，共 (log n\) 层，故 T(n) = O(n log n)。

- **代入法**：

     现有进程：P0、P1（共 2 个进程）。

     每个进程调用`fork()`，各自创建 1 个子进程（共新增 2 个进程），此时总进程数为 4。

     这 4 个进程都会执行`print("-")`，共打印 **4 个`-`**。

     这 8 个进程都会执行`print("-")`，共打印 **8 个`-`**。

   - **`i=3`时**：

     现有进程：8 个。

     每个进程调用`fork()`，新增 8 个进程，总进程数为 16。

     这 16 个进程都会执行`print("-")`，共打印 **16 个`-`**。

总打印数量：每次循环的打印数依次为`2、4、8、16`，总和为：2+4+8+16=30

## 5对括号的合法组合数量

要计算5对括号的合法组合数量，我们需要找到所有由5个左括号"("和5个右括号")"组成的序列，且满足在任意前缀中，左括号数量不少于右括号数量。这类问题通常用**卡特兰数**解决，其公式为：

$$
C_n = \frac{1}{n+1} \binom{2n}{n}
$$


对于 n=5，有 C5=42。下面介绍几种思考方法，帮助你理解这个结果。

---

### 方法一：递归分解（分治思想）

合法括号序列有一个重要性质：第一个字符必须是左括号，并且总存在一个右括号与之匹配。假设第一个左括号与第 k 个右括号配对（这里的 kk 表示该右括号在序列中的位置），那么这一对括号将整个序列分成两部分：

- **内部**：从第2个字符到第 k−1 个字符，是一个合法的括号序列，设其有 i 对括号。
- **外部**：从第 k+1 个字符到结尾，也是一个合法的括号序列，有 n−1−i 对括号。

由于内部和外部独立，总方案数就是所有可能的 i 对应的内部方案数乘以外部的方案数之和。即递推式：

$$
f(n) = \sum_{i=0}^{n-1} f(i) \cdot f(n-1-i), \quad f(0)=1
$$

对于 n=5，我们可以从 n=0 开始逐步计算：

$$
\begin{align*}
f(0) &= 1 \quad \text{（空序列）} \\
f(1) &= f(0) \cdot f(0) = 1 \\
f(2) &= f(0)f(1) + f(1)f(0) = 1 + 1 = 2 \\
f(3) &= f(0)f(2) + f(1)f(1) + f(2)f(0) = 2 + 1 + 2 = 5 \\
f(4) &= f(0)f(3) + f(1)f(2) + f(2)f(1) + f(3)f(0) = 5 + 2 + 2 + 5 = 14 \\
f(5) &= f(0)f(4) + f(1)f(3) + f(2)f(2) + f(3)f(1) + f(4)f(0) = 14 + 5 + 4 + 5 + 14 = 42
\end{align*}
$$

因此，5对括号有42种合法组合。

---

### 方法二：组合数学与反射法

如果不考虑合法性，仅从10个位置中选择5个放左括号，总共有 (10 5)=252种方式。但其中很多序列是不合法的，例如前几个字符中右括号过多。我们可以用**反射法**计算合法数量。

将左括号视为向上移动一步（+1），右括号视为向下移动一步（-1），那么一个序列对应一条从 (0,0) 到 (10,0)的折线。合法性要求折线始终不越过x轴（即高度非负）。所有路径中，第一次到达负值的路径可以通过反射变换映射为一条从 (0,0)到 (10,−2) 的路径，这种路径的数量等于 (10 4)（因为需要4个上升和6个下降？注意：到 (10,−2) 意味着上升比下降少2，设上升数为 u，下降数为dd，则 u+d=10，u−d=−2 得 u=4,d=6，所以路径数为 (10 4）。因此，合法路径数等于总路径减去非法路径：

$$
\binom{10}{5} - \binom{10}{4} = 252 - 210 = 42
,而
\binom{10}{4} = \binom{10}{6}
$$

## 问题：Prim算法

Prim算法是一种用于求解加权无向图的最小生成树（MST，Minimum Spanning Tree）的贪心算法。它通过逐步扩展一棵树，每次添加一条连接树中顶点与树外顶点的最小权重边，直到所有顶点都在树中。

---

### 算法思想

1. **初始化**：选择一个起始顶点，将其加入生成树集合 UU。
2. **重复以下步骤直到 UU 包含所有顶点**：
    
    - 在所有连接 UU 中顶点与 V∖UV∖U 中顶点的边中，选择权重最小的边 (u,v)(u,v)，其中 u∈Uu∈U，v∉Uv∈/U。
    - 将顶点 vv 加入 UU，并将边 (u,v)(u,v) 加入生成树。

---

### 算法步骤（以邻接矩阵或邻接表为例）

常用实现方式有两种：朴素实现（适用于稠密图）和优先队列优化（适用于稀疏图）。

### 朴素实现（时间复杂度 O(V2)O(V2)）

- 使用两个数组：
    - `key[i]`：表示顶点 i 到当前生成树的最小边权（初始为无穷大，起始点设为0）
    - `parent[i]`：记录生成树中顶点 i 的父节点。
    - `visited[i]`：标记顶点是否已在树中。
- 每次从未访问顶点中选择 `key` 值最小的顶点加入树，并更新其邻接点的 `key` 值。

### 优先队列优化（时间复杂度 O(Elog⁡V)O(ElogV)）

- 使用最小堆（优先队列）存储每个未加入树的顶点及其当前最小边权。
- 每次取出堆顶顶点加入树，然后遍历其邻边，如果邻接点未在树中且边权更小，则更新其 `key` 值并压入堆。


### 复杂度分析

- **邻接矩阵 + 朴素查找**：O(V2)O(V2)，适合稠密图。
- **邻接表 + 二叉堆**：O(Elog⁡V)O(ElogV)，适合稀疏图。
- **邻接表 + 斐波那契堆**：O(E+Vlog⁡V)O(E+VlogV)，理论更优但实现复杂。



### 与Kruskal算法的比较

- Prim算法从顶点出发，适合边稠密的图。
- Kruskal算法从边出发，适合边稀疏的图。
- 两者都是贪心算法，都能正确得到最小生成树。

## 5.1  thread_local 关键字

### 核心定义

`thread_local` 是 C++11 引入的**存储期说明符**，用于声明变量的生命周期与线程绑定：

- 每个线程拥有该变量的**独立副本**，线程间互不干扰；
- 变量在线程创建时初始化，线程结束时销毁；
- 可与 `static`/`extern` 结合使用（但优先级高于二者）。

### 典型示例

```cpp
#include <iostream>
#include <thread>
using namespace std;

thread_local int num = 0; // 每个线程有独立的num

void func(int id) {
    num = id; // 每个线程修改自己的副本
    cout << "线程" << id << "：num = " << num << endl;
}

int main() {
    thread t1(func, 1);
    thread t2(func, 2);
    t1.join();
    t2.join();
    return 0;
}
```

### 输出结果

```
线程1：num = 1
线程2：num = 2
```

### 常见使用场景

- 线程内需要全局共享、但线程间隔离的变量（如线程专属的日志缓冲区、临时计算变量）；
- 替代线程局部存储（TLS）的平台相关 API（如 Windows 的`TlsSetValue`），实现跨平台。

## 5.2 软链接（符号链接）和硬链接的区别

表格

|跨文件系统|不支持（inode 是文件系统内唯一）|支持|
|指向目录|不允许（避免目录循环引用）|允许|
|原文件删除|仍可访问（链接数 > 0 则文件数据存在）|失效（变成 “断链”）|
|权限|与原文件一致（权限是文件属性）|权限为 lrwxrwxrwx（仅表示链接本身）|

### 关键补充

- 硬链接通过 `ln 源文件 硬链接文件` 创建；
- 软链接通过 `ln -s 源文件 软链接文件` 创建；
- 查看 inode：`ls -i 文件名`。

## 5.3 判断回文字符串（输入 char* ptr）

### 核心思路

- 先处理边界（空指针、空字符串）；
- 双指针法：左指针从头部、右指针从尾部向中间遍历，逐一比较字符；
- 忽略大小写 / 空格（可选，根据需求调整）。

### 完整代码

```cpp
#include <cstdio>
#include <cstring>
#include <cctype> // 用于tolower

// 判断是否为回文字符串，输入char* ptr
bool isPalindrome(char* ptr) {
    // 边界条件：空指针或空字符串
    if (ptr == nullptr || *ptr == '\0') {
        return true; // 定义空字符串为回文（可根据需求调整）
    }

    // 双指针初始化
    char* left = ptr;
    char* right = ptr + strlen(ptr) - 1;

    // 向中间遍历比较
    while (left < right) {
        // 跳过非字母数字（可选，如仅判断字母数字）
        while (left < right && !isalnum(*left)) {
            left++;
        }
        while (left < right && !isalnum(*right)) {
            right--;
        }

        // 统一转为小写比较（忽略大小写）
        if (tolower(*left) != tolower(*right)) {
            return false;
        }
        left++;
        right--;
    }
    return true;
}

// 测试用例
int main() {
    char str1[] = "abba";
    char str2[] = "abcba";
    char str3[] = "A man, a plan, a canal: Panama";
    char str4[] = "hello";
    char* str5 = nullptr;

    printf("\"%s\": %s\n", str1, isPalindrome(str1) ? "是回文" : "不是回文");
    printf("\"%s\": %s\n", str2, isPalindrome(str2) ? "是回文" : "不是回文");
    printf("\"%s\": %s\n", str3, isPalindrome(str3) ? "是回文" : "不是回文");
    printf("\"%s\": %s\n", str4, isPalindrome(str4) ? "是回文" : "不是回文");
    printf("空指针: %s\n", isPalindrome(str5) ? "是回文" : "不是回文");

    return 0;
}
```

### 输出结果

```
"abba": 是回文
"abcba": 是回文
"A man, a plan, a canal: Panama": 是回文
"hello": 不是回文
空指针: 是回文
```

## 5.4 STL 容器及线程安全性

### 5.4.1 STL 核心容器分类

| 容器类型    | 具体容器                                                                   |
| ------- | ---------------------------------------------------------------------- |
| 序列式容器   | vector、list、deque、array、forward_list                                   |
| 关联式容器   | set/multiset、map/multimap（红黑树实现）                                       |
| 无序关联式容器 | unordered_set/unordered_multiset、unordered_map/unordered_multimap（哈希表） |
| 容器适配器   | stack、queue、priority_queue                                             |
| 字符串容器   | string（本质是 char 的序列容器）                                                 |
| 数组容器    | bitset（固定大小）、valarray（数值数组）                                            |

### 5.4.2 线程安全性

**结论：C++ 标准 STL 容器默认** 不保证线程安全 **，具体规则：**

1. **只读操作**：多个线程同时读取同一个容器 → 安全；
2. **写操作**：至少一个线程写、其他线程读 / 写 → 不安全（会导致数据竞争）；
3. **例外**：
    
    - C++17 起，`std::string` 的某些只读操作（如`c_str()`）保证线程安全；
    - 部分编译器 / 库（如 MSVC）对特定容器做了弱线程安全优化，但不跨平台；
    
4. **解决方案**：
    
    - 手动加锁（`std::mutex`/`std::lock_guard`）；
    - 使用线程安全的容器替代（如`boost::thread_safe_container`）；
    - 读写分离（`std::shared_mutex`）。
    

## 5.5 结构体中含 static 成员时的 sizeof 结果

### 核心规则

- `static` 成员属于**类 / 结构体**，而非**对象实例**，因此不计入对象的内存布局；
- `sizeof(结构体)` 仅计算**非静态成员**的大小（含内存对齐）。

### 示例验证

```cpp
#include <iostream>
using namespace std;

struct Test {
    char c;       // 1字节（对齐后通常占4字节）
    int i;        // 4字节
    static long s; // 静态成员，不计入sizeof
};
long Test::s = 10; // 静态成员类外初始化

int main() {
    // 结果：8字节（char占1字节，对齐补3字节 + int4字节）
    cout << "sizeof(Test) = " << sizeof(Test) << endl;
    return 0;
}
```

### 补充说明

- 内存对齐：编译器会自动对齐成员变量（如 char 后补 3 字节，使 int 按 4 字节对齐）；
- 空结构体：`sizeof(空结构体)` 为 1 字节（编译器为其分配唯一地址）；
- 静态成员存储在全局 / 静态存储区，与对象实例无关。
